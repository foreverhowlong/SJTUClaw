# 展示大纲

## 任务契约

开始前整理：展示场景、受众基础、总时长、目标、必须覆盖的内容、可用材料、是否包含问答，以及用户需要的是页面结构、讲稿提纲还是两者兼有。

- 缺少总时长或受众信息且会明显影响结构时，优先追问。
- 只使用用户提供或实际读取的事实；不为“建议视觉”虚构数据、图表或案例。

## 叙事设计

1. 先写出一句话展示目标：听众结束后应该理解、相信或决定什么。
2. 建立一条清晰叙事线，通常包含“为什么重要—问题是什么—证据或方案—意味着什么—下一步”。根据展示类型调整，不机械套用。
3. 先分配章节时间，再决定页数。确保各页时长之和不超过总时长，并为开场、总结、切换和问答留出空间。
4. 每页只承担一个明确目的和一个主要结论。不要把报告段落直接搬到幻灯片，也不要用大量项目符号代替结构。

## 逐页格式

每页使用以下字段，内容保持简洁、可直接制作：

```markdown
## 01｜页面标题（预计 00:45）

- **页面目的：** 这页要推进叙事中的哪一步
- **核心信息：** 听众离开这页时应记住的一句话
- **页面内容：** 最多 3–5 个必要信息点
- **视觉建议：** 图表、流程、对比、引用或留白；说明视觉服务于什么信息
- **讲稿要点：** 适合口头表达的提示，不写成逐字稿
- **转场：** 如何自然进入下一页
```

## 质量检查与交付

1. 检查开场是否快速建立问题和价值，主体是否有证据支撑，结尾是否给出明确结论或行动。
2. 检查时间总和、页面密度和转场；删去不能推进叙事的页面，而不是压缩所有页面的讲述时间。
3. 视觉建议必须具体且可实现。没有数据时可以建议结构图或概念图，但不得暗示存在未提供的统计结果。
4. 用户要求保存时，使用 `create_file`、`overwrite_file` 或 `edit_file` 写入 Markdown 大纲，并经过现有 approval；工具成功前不得声称已经保存。
5. 最终答复概括叙事线、总页数、时间分配、实际保存路径，以及仍需补充的材料。
